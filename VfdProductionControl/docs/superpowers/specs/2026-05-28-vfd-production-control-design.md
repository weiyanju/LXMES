# VFD Production Control 重构需求设计

日期：2026-05-28
设计语言修订：2026-06-04

## 1. 背景与目标

VfdProductionControl 是面向产线工位的 VFD 测试执行与监控系统。当前项目已经具备 WPF 重构基础，Phase 1 的目标是做出“模拟优先的可试运行版本”：先跑通技工生产操作、工程师方案配置、多槽位并行测试、仪表对照校验、结果追溯和本地数据保存，再接入真实串口、真实仪表和外部生产系统。

本次文档重写将产品设计语言调整为 iOS HIG 的工业桌面转译：清晰层级、有限色彩、可读字号、适度圆角、轻量材料感、即时反馈和可访问状态表达。它不是手机 App 化，也不是玻璃拟态化，而是让一套 Windows WPF 工业软件拥有更现代、更克制、更一致的交互品质。

## 2. 产品定位

系统定位为流水线变频器测试方案执行与监控平台，而不是通用 Modbus 调试工具。

主要用户：

- **技工**：在工位安装变频器，扫描员工码和 VFD 条码，选择方案和槽位，执行测试并查看结论。
- **工程师**：维护测试方案、步骤、命令、规则、容差、失败策略和模拟验证。
- **管理员**：维护账号、员工码、工位、槽位、串口、仪表、条码规则和系统配置。
- **质量与追溯用户**：查询历史会话、槽位运行、步骤、测量、比较、命令和异常记录。

第一阶段仍保留工程调试能力，但调试能力必须收纳在工程维护区域，并受权限控制，不进入技工的日常操作台。

## 3. 设计语言

### 3.1 设计北极星

**iOS-HIG Industrial Console**

界面应该像一台明亮、稳定、可信的现代产线控制台：信息密度高但不拥挤，状态醒目但不刺眼，动作明确但不喧宾夺主。操作员进入系统后，首先看到的是当前工位、槽位、下一步动作和测试状态，而不是配置复杂度。

### 3.2 HIG 转译原则

- **Hierarchy 层级**：导航、当前任务、主数据、次级详情分层展示。当前生产状态永远高于辅助信息。
- **Harmony 和谐**：圆角、留白、字号、状态标签、输入和按钮形成同一套视觉节奏。
- **Consistency 一致**：四个模块共享组件语言。相同状态在不同页面必须长得一样、叫法一样。
- **Color 色彩**：使用有限色彩。蓝色表示交互和焦点，绿色/黄色/红色表示生产结论与异常。
- **Typography 字体**：使用系统字体和稳定数据字体，通过字重、字号和分组建立层级。
- **Materials 材料**：用浅色表面、细边框和轻阴影表现层级。不要把内容区做成半透明玻璃。
- **Accessibility 可访问性**：状态不只靠颜色，关键控件尺寸适合反复点击，文本对比度满足明亮产线环境。

### 3.3 视觉规则

- 主背景为冷灰白，面板为白色或近白色。
- 互动主色为 iOS 风格蓝：用于主按钮、选中模块、焦点和可点击强调。
- 通过、警告、失败、运行中使用语义色，并始终配中文状态文本。
- 圆角以 6、8、10 px 为主，不使用大圆角营销卡片。
- 控件高度：普通动作不低于 40 px，生产关键动作不低于 44 px。
- 数据值、计时、COM 口、寄存器和测量值可使用等宽字体。
- 不使用大面积渐变、霓虹色、装饰性玻璃、重复方块卡片或纯图形装饰。

## 4. 技术与部署约束

- 桌面技术：WPF + .NET。
- 数据库方向：中央 SQL Server。
- 部署形态：多工位 WPF 客户端 + 中央数据库。
- 第一阶段通信：模拟设备优先。
- 后续真实通信：每个槽位通过独立串口连接一台变频器和一组监控仪表。
- 旧项目仅作为通信经验和部分实现参考，不在旧目录中继续堆叠新版本代码。

架构目标：

- 领域模型和执行规则独立于 UI。
- ViewModel 暴露适合界面展示的状态，不让 XAML 推导业务结论。
- 模拟、串口、数据库和外部系统通过接口隔离。
- 历史数据保存执行时快照，避免后续配置修改改变历史解释。

## 5. 核心生产流程

第一阶段技工流程：

1. 技工扫描员工码进入生产会话。
2. 选择本次要执行的测试方案。
3. 系统显示当前工位可用槽位。
4. 技工选择参与测试的槽位。
5. 按所选槽位顺序扫描 VFD 条码。
6. 系统按配置决定是否展示开始前确认。
7. 技工确认后，所有选中槽位并行执行同一方案版本。
8. 每个槽位拥有独立状态、步骤进度、读数、比较结果和最终结论。
9. 测试完成后，界面显示槽位结论并保存完整追溯数据。

典型测试步骤：

1. 软件发送变频器启动指令。
2. 读取运行状态或关键参数，确认变频器已运行。
3. 等待稳定时间，例如 5 秒。
4. 读取变频器电压。
5. 读取仪表电压。
6. 计算变频器电压与仪表电压的差值。
7. 按配置容差生成小结论。
8. 继续执行后续步骤。
9. 测试结束或异常收尾时发送停止指令。
10. 读取停止状态或关键参数，确认设备已停止。

外部启动第一阶段不处理。启动和停止均由软件通过配置好的命令完成。

## 6. 工位、槽位与通信模型

系统把工位、槽位和串口解耦。

核心对象：

- `Station`：工位，例如产线 1 号测试工位。
- `StationSlot`：工位槽位，例如 1 号位、2 号位。
- `SlotCommunicationConfig`：槽位当前通信配置，包括串口名、波特率、数据位、校验位、停止位和超时。
- `SlotPrimaryDevice`：槽位主设备，第一阶段为变频器。
- `SlotInstrument`：槽位绑定的一组监控仪表。
- `InstrumentPoint`：仪表点位，例如电压、电流、功率。

约束：

- 一个工位可以有多个槽位。
- 一个槽位对应一台变频器和一组仪表。
- 每个槽位当前绑定一个串口，例如 COM1。
- COM1 是配置值，不是槽位身份。同一槽位后续可改为 COM5 或其他串口。
- 历史记录必须保存执行时的槽位、串口和通信配置快照。
- 同一槽位内访问变频器和仪表时，Modbus 请求串行。
- 不同槽位之间可以并行执行。

## 7. 仪表对照校验

监控仪表不是单独做上下限判定的主对象。仪表用于实时监测变频器实际输出，并与变频器自身读数做对照，判断变频器读数是否正常。

示例：

- 从变频器读取上报电压。
- 从电压仪表读取实测电压。
- 系统计算差值和偏差百分比。
- 差值在允许误差内，则该对照项通过。
- 差值超出允许误差，则按规则生成 `Warning` 或 `Fail`。

判定模型：

- `PrimaryMeasurement`：变频器侧读数。
- `ReferenceMeasurement`：仪表侧实测值。
- `RangeRule`：对变频器读数做上下限判定。
- `ComparisonRule`：对变频器读数与仪表读数做偏差判定。
- `ToleranceType`：`Absolute` 或 `Percent`。
- `ToleranceValue`：允许误差值。
- `ComparisonResult`：正常、警告、失败、仪表不可用、数据无效。

容差要求：

- 支持绝对误差，例如 `±2V`。
- 支持百分比误差，例如 `±1%`。
- 工程师在规则配置中选择容差类型。
- 百分比误差默认以仪表实测值为基准。

追溯记录保存：

- 变频器原始值和换算值。
- 仪表原始值和换算值。
- 差值和偏差百分比。
- 容差类型和值。
- 小结论。
- 判定规则快照。

## 8. 方案与步骤模型

核心对象不是“变频器永久结论”，而是“一名员工在某个工位槽位，对某个 VFD 条码执行某个方案版本的一次测试记录”。

建议模型：

- `ProcessPlan`：测试方案主数据。
- `ProcessPlanVersion`：方案版本。工程师每次保存生成新版本，可启用、停用和回滚。
- `ProcessStep`：方案步骤，支持分组、顺序、读取、写入、延时、确认、启动、停止、条件分支。
- `StepCommand`：具体命令定义，包括设备来源、功能码、地址、长度、数据类型、缩放、单位。
- `StepRule`：判定规则。
- `StepFailurePolicy`：失败处理策略。
- `StationSession`：工位生产会话。
- `DeviceRun`：某个槽位上某台 VFD 的一次方案执行。
- `StepRun`：某台 VFD 某个步骤的执行记录。
- `CommandTrace`：单条通信请求和响应记录。

步骤类型至少包括：

- 软件启动变频器。
- 软件停止变频器。
- 运行状态确认。
- 读取变频器数值。
- 读取仪表数值。
- 读取变频器或仪表字符串。
- 写变频器参数。
- 延时等待。
- 人工确认。
- 数据比对。
- 条件判断。

## 9. 数值与字符串判定

步骤判定规则支持数值和字符串两类。

数值判定：

- 上限和下限。
- 绝对误差。
- 百分比误差。
- 变频器读数与仪表读数交叉比对。
- 单位、缩放、精度。

字符串判定：

- 等于指定值。
- 不等于指定值。
- 包含指定内容。
- 不包含指定内容。
- 前缀匹配。
- 后缀匹配。
- 正则匹配。
- 多个允许值列表。
- 大小写敏感可配置。

字符串来源：

- 变频器型号。
- 固件或软件版本号。
- 状态文本。
- 条码或序列号。
- 参数组编号。
- 仪表返回的字符串。

第一阶段保留字符串读取和解析配置能力。模拟器按配置返回字符串，用于流程验证。真实解析规则后续根据现场设备协议完善。

## 10. 小结论、总结构与槽位展示

测试项目由多个步骤和多个小结论组成。

规则：

- 每个判定项生成一个小结论。
- 小结论可以是 `Pass`、`Fail` 或 `Warning`。
- 只有所有会影响结果的小结论都合格，槽位本次测试总结构才合格。
- `Warning` 不等于失败，但必须高亮并保存。
- 如果某个小结论为 `Fail`，槽位总结构为 `Fail`，除非该规则配置为只警告。

槽位视觉状态：

- `通过`：绿色，带对勾或通过文本。
- `失败`：红色，带叉号或失败文本。
- `警告`：黄色，带警告图标或警告文本。
- `运行中`：靛蓝或蓝色，带进度和步骤名。
- `待扫码`、`未开始`、`已移除`：中性色。

槽位模块直接展示关键数据：

- 槽位号。
- COM 口。
- VFD 条码。
- 当前步骤。
- 当前进度。
- 变频器电压。
- 仪表电压。
- 差值。
- 小结论。
- 主要异常提示。

详细数据进入详情页或展开区，避免槽位模块过载。

## 11. 失败策略与收尾停止

失败处理必须是步骤级和规则级可配置。

建议策略：

- `ContinueAndMarkFail`：继续执行后续步骤，但槽位最终失败。
- `ContinueAsWarning`：继续执行，仅记录警告。
- `StopSlotImmediately`：立即停止当前槽位变频器并终止该槽位流程。
- `PauseAllSlots`：暂停全部槽位，等待人工处理。
- `RetryThenStop`：重试 N 次后停止当前槽位。
- `RequireOperatorConfirm`：要求技工确认后继续或停止。

安全收尾：

- 正常完成后必须发送停止指令。
- 步骤失败且策略要求停止时必须发送停止指令。
- 人工中止时，如果通信可用，也应尝试发送停止指令。
- 停止指令执行结果和停止确认结果必须保存。
- 停止指令是收尾流程的一部分，避免设备保持运行。

## 13. 开始前检查

点击开始后，正式测试前执行安全检查。

检查项：

- 选中槽位是否都有 VFD 条码。
- 方案版本是否可执行。
- 槽位串口配置是否完整。
- 模拟模式或真实通信模式是否可用。
- 变频器是否响应。
- 仪表是否响应。
- 数据库连接是否可用。

开始前确认页可配置：

- 开启时，扫完条码后显示确认页。
- 关闭时，扫完最后一个条码后自动开始。

确认页显示：

- 员工码。
- 方案名称和版本。
- 所选槽位。
- 槽位对应条码。
- 槽位当前串口配置。
- 仪表组摘要。

## 14. 数据存储策略

建议采用结构化字段 + JSON 明细的混合存储方式。

用于查询、筛选、统计的核心信息使用结构化字段：

- 员工码。
- 工位。
- 槽位。
- 串口配置快照摘要。
- VFD 条码。
- 测试方案和版本。
- 开始时间。
- 结束时间。
- 总结论。
- 步骤编号和名称。
- 小结论。
- 变频器读数。
- 仪表读数。
- 差值。
- 偏差百分比。
- 上限和下限。
- 容差类型和值。
- 异常类型。

扩展和追溯细节使用 JSON：

- 原始 Modbus 请求和响应。
- 多寄存器原始值。
- 换算参数。
- 判定规则快照。
- 步骤配置快照。
- 仪表点位配置快照。
- 重试过程。
- 额外诊断信息。

这样既方便查询“不合格电压测试”等常见问题，也能完整保留现场追溯上下文。

## 16. 通信与模拟模式

第一阶段默认模拟模式。

通信接口建议：

- `IBarcodeInputService`：扫码输入源，支持键盘扫码枪和串口扫码枪。
- `IDeviceTransport`：底层设备传输抽象。
- `IDeviceCommandClient`：面向步骤执行的读写接口。
- `ISimulatedDeviceClient`：模拟设备实现。
- `ICommandTraceWriter`：请求、响应、耗时和异常记录。

模拟器能力：

- 按槽位模拟变频器。
- 按槽位模拟多个仪表。
- 模拟启动、停止、读数、字符串返回。
- 模拟通信超时、设备拒绝、仪表故障、值超限、字符串不匹配。
- 支持配置测试数据，验证方案和 UI。

真实通信预留：

- 每个槽位一个串口。
- 每个槽位串口下有一台变频器和多个仪表。
- 同一槽位串口内部请求串行。
- 不同槽位串口之间并行。

## 17. 权限与入口

角色：

- **技工**：扫码员工码进入生产会话，选择方案和槽位，扫描条码，执行测试，查看当前结果。
- **工程师**：账号密码登录，维护方案、步骤、规则、失败策略，执行模拟验证和工程调试。
- **管理员**：账号密码登录，维护账号、员工码、工位、槽位、扫码规则、串口配置和系统设置。

入口：

- 默认进入操作员控制台。
- 工程维护、系统管理、追溯查询通过顶部导航进入。
- 工程调试功能不对技工显示。

临时条码规则：

- 员工码：`EMP` + 4 到 8 位数字，例如 `EMP0001`。
- VFD 条码：`VFD` + 8 到 20 位数字或大写字母，例如 `VFD202605280001`。
- 后续可由管理员配置正则、前缀和长度。

## 18. 页面设计

页面设计遵循“操作员中心，工程能力收纳，追溯可解释”的结构。

### 操作员控制台

操作员控制台是主界面。

区域：

- 顶部全局导航和工位状态。
- 流程进度：员工码、方案、槽位、条码、确认、运行、完成。
- 槽位看板：显示每个槽位的状态、条码、当前步骤、测量和结论。
- 当前动作面板：只突出下一步主动作。
- 运行明细和指令日志：显示步骤、测量、比较、命令和异常。

设计要求：

- 运行时槽位看板不可被配置页面替代。
- 主按钮最多一个。
- 状态标签必须带中文。
- 条码和员工码输入适合扫码枪快速录入。

## 21. 第一阶段成功标准

第一阶段完成后应能演示：

1. 管理员配置工位、槽位、串口和仪表。
2. 工程师创建测试方案，配置启动、等待、读取、比对、停止等步骤。
3. 工程师保存方案版本并设为可执行。
4. 技工扫描员工码进入会话。
5. 技工选择方案。
6. 技工选择多个槽位。
7. 技工按槽位顺序扫描多个 VFD 条码。
8. 系统显示确认页或按配置自动开始。
9. 多个槽位并行执行模拟测试。
10. 每个槽位显示关键读数、小结论和总结构。
11. 某个槽位失败不影响其他槽位继续，除非策略要求暂停全部。
12. 测试完成或异常终止时，系统尝试发送停止指令。
13. 历史查询可以查看会话、槽位、条码、步骤、读数、比较结果、请求/响应和异常。
14. UI 使用统一的顶部导航、槽位看板、当前动作面板、状态标签和中文文案。
15. 状态表达不只依赖颜色，且在明亮工位环境中可读。
