# LXMES 条码规则设计文档

## 1. 设计结论

LXMES 的条码体系分为两组概念：

```text
主条码 / 子条码：描述产品组成关系
内部条码 / 客户条码：描述对内追溯和对外展示
```

这两组概念不能互相替代。

VFD 测试扫码入口明确为：

```text
变频器内部主条码
```

也就是说，WPF 操作端在 VFD 测试开始时扫描的是 MES 内部主条码，不是客户条码，也不是子条码。

## 2. 当前涉及的已有表

### 2.1 生产计划 `production_plans`

当前生产计划保存两套条码范围：

| 字段 | 含义 | 用途 |
| --- | --- | --- |
| `start_barcode` | 内部主条码开始范围 | 内部生产、工单匹配、追溯 |
| `end_barcode` | 内部主条码结束范围 | 内部生产、工单匹配、追溯 |
| `customer_start_barcode` | 客户条码开始范围 | 客户展示、交付资料、对外查询 |
| `customer_end_barcode` | 客户条码结束范围 | 客户展示、交付资料、对外查询 |

生产计划的内部条码范围表示该计划覆盖的内部主条码总范围。

客户条码范围表示该计划对应的客户展示条码总范围。

### 2.2 工单 `work_orders`

当前工单也保存两套条码范围：

| 字段 | 含义 | 用途 |
| --- | --- | --- |
| `start_barcode` | 内部主条码开始范围 | 内部扫码定位工单 |
| `end_barcode` | 内部主条码结束范围 | 内部扫码定位工单 |
| `customer_start_barcode` | 客户条码开始范围 | 工单级客户展示 |
| `customer_end_barcode` | 客户条码结束范围 | 工单级客户展示 |

一个生产计划可以拆分多个工单。工单的内部条码范围应该落在生产计划内部条码范围内。

工单的客户条码范围应该落在生产计划客户条码范围内。

### 2.3 主子条码关系 `barcode_relations`

`barcode_relations` 继续承载内部主条码和内部子条码的组成关系：

| 字段 | 含义 |
| --- | --- |
| `main_barcode` | 内部主条码 |
| `main_barcode_type` | 主条码类型 |
| `sub_barcode` | 内部子条码 |
| `sub_barcode_type` | 子条码类型 |
| `product_id` | 产品 |
| `work_order_id` | 工单 |

示例：

```text
内部主条码：LX-VFD-000001
内部子条码：外壳条码、主板条码、电源板条码、通讯板条码
```

`barcode_relations` 不建议用于表达内部条码和客户条码的一一映射。它的职责是产品组成关系，不是客户展示编号映射。

### 2.4 扫描记录 `barcode_scans`

`barcode_scans.barcode` 记录某次工序扫描的条码。

对于 VFD 测试工序：

```text
barcode_scans.barcode = 变频器内部主条码
```

`barcode_scans` 可以记录“进入 VFD 测试工序”“完成 VFD 测试工序”等事件，但不保存每个测试步骤、通信报文或测量明细。

## 3. 条码类型边界

### 3.1 内部条码编码格式

一期内部条码统一采用无分隔符、固定长度、可解析结构：

```text
LXVFD + 角色位 + 部件类型位 + 6 位流水号
```

字段拆分：

| 位置 | 字段 | 示例 | 说明 |
| --- | --- | --- | --- |
| 第 1-2 位 | 公司/系统前缀 | `LX` | 利翔 / LXMES |
| 第 3-5 位 | 产品/业务前缀 | `VFD` | 变频器业务 |
| 第 6 位 | 条码角色位 | `1` / `2` | `1` 主条码，`2` 子条码 |
| 第 7 位 | 部件类型位 | `1`-`9` | 整机或部件类型 |
| 第 8-13 位 | 流水号 | `000001` | 固定 6 位数字 |

示例：

| 条码 | 角色 | 部件类型 | 说明 |
| --- | --- | --- | --- |
| `LXVFD11000001` | 主条码 | 整机/变频器本体 | VFD 测试扫码入口 |
| `LXVFD22010325` | 子条码 | 外壳 | 绑定到内部主条码 |
| `LXVFD23005210` | 子条码 | 主板 | 绑定到内部主条码 |
| `LXVFD24001888` | 子条码 | 电源板 | 绑定到内部主条码 |
| `LXVFD25000666` | 子条码 | 通讯板 | 绑定到内部主条码 |

条码中不使用 `-` 等分隔符。旧示例 `LX-VFD-000001` 不作为一期正式格式。

角色位约定：

| 角色位 | 含义 |
| --- | --- |
| `1` | 内部主条码 |
| `2` | 内部子条码 |

部件类型位约定：

| 部件类型位 | 含义 |
| --- | --- |
| `1` | 整机/变频器本体 |
| `2` | 外壳 |
| `3` | 主板 |
| `4` | 电源板 |
| `5` | 通讯板 |
| `6` | 继电器板 |
| `7` | 驱动板 |
| `8` | 键盘板 |
| `9` | 预留 |

系统校验应同时使用“条码本身编码”和“数据库条码类型字段”：

```text
条码角色位 = 1，数据库类型应为主条码/厂内条码
条码角色位 = 2，数据库类型应为子条码类型
```

### 3.2 内部主条码

内部主条码是 MES 内部生产、测试、追溯的核心条码。

用途：

- 匹配工单
- 匹配生产计划
- 进入工序扫描
- VFD 测试入口
- 追溯查询主键
- 关联内部子条码

VFD 测试中，内部主条码是唯一的扫码入口。

编码规则：

```text
内部主条码必须匹配 LXVFD11 + 6 位数字
```

示例：

```text
LXVFD11000001
LXVFD11000002
LXVFD11000003
```

VFD 测试时如果扫描到非 `LXVFD11` 格式的条码，应提示“请扫描变频器内部主条码”。

### 3.3 内部子条码

内部子条码表示整机或变频器下挂的组成件条码。

例如：

- 外壳条码
- 主板条码
- 电源板条码
- 通讯板条码
- 继电器板条码
- 驱动板条码
- 键盘板条码

内部子条码通过 `barcode_relations.main_barcode -> barcode_relations.sub_barcode` 关联到内部主条码。

编码规则：

```text
内部子条码必须匹配 LXVFD2 + 部件类型位 + 6 位数字
```

示例：

```text
LXVFD22010325  外壳子条码
LXVFD23005210  主板子条码
LXVFD24001888  电源板子条码
```

子条码不参与工单范围匹配，也不作为 VFD 测试入口。

主条码和子条码的流水号不要求一致。实际主子关系以 `barcode_relations` 绑定记录为准，不通过流水号推导。

### 3.4 客户条码

客户条码是对客户展示、交付、查询使用的条码。

用途：

- 标签展示
- 发货资料
- 客户查询
- 对外报表

客户条码不作为内部工单匹配和 VFD 测试入口。

客户条码和内部主条码在业务上是一一对应关系，但具体对应方式后续单独设计。

## 4. VFD 测试条码流程

VFD 测试扫码流程如下：

```text
扫描变频器内部主条码
-> 解析内部主条码
-> 根据 work_orders.start_barcode / work_orders.end_barcode 匹配工单
-> 得到 work_order
-> 得到 production_plan
-> 得到 product_id
-> 找到 VFD 测试工序
-> 找到 VFD 测试方案
-> 查询 barcode_relations 获取内部子条码
-> 执行 VFD 测试
-> 测试追溯绑定内部主条码
-> 客户展示时再通过客户条码映射展示
```

VFD 测试追溯表中应该保存内部主条码。

客户条码可以作为展示字段或后续映射查询结果返回，但不应替代内部主条码。

## 5. 条码范围匹配规则

当前内部条码规则按以下格式设计：

```text
LXVFD + 角色位 + 部件类型位 + 6 位流水号
```

示例：

```text
LXVFD11000001
LXVFD11010000
```

范围匹配不建议直接使用字符串大小比较。

推荐由后端独立服务负责解析和匹配：

```text
BarcodeRangeMatcher
```

职责：

- 解析条码前缀
- 解析角色位
- 解析部件类型位
- 解析数字流水号
- 校验前缀是否一致
- 校验角色位和部件类型位是否符合场景要求
- 校验数字位数是否一致
- 判断扫描条码是否落在开始和结束范围内
- 根据内部主条码定位工单和生产计划

这样后续条码规则变动时，只需要调整条码解析服务，不需要改 VFD 测试、工单、生产计划和追溯主流程。

## 6. 客户条码和内部主条码映射

业务原则：

```text
一个内部主条码 对应 一个客户条码
一个客户条码 对应 一个内部主条码
```

当前阶段先不在 `barcode_relations` 中表达这个映射。

原因：

- `barcode_relations` 的职责是内部主子条码组成关系。
- 客户条码是对外展示编号，不是产品内部组成件。
- 客户条码规则后续可能变化，过早放入主子条码表会污染原有语义。

后续如需精确表达单件映射，建议新增独立映射表，例如：

```text
barcode_mappings
- internal_main_barcode
- customer_barcode
- work_order_id
- product_id
- mapping_source
- status
- created_at
- updated_at
```

该表只负责内部主条码和客户条码的一一映射。

## 7. 字段使用边界

### 7.1 内部流程优先使用

```text
production_plans.start_barcode
production_plans.end_barcode
work_orders.start_barcode
work_orders.end_barcode
barcode_relations.main_barcode
barcode_relations.sub_barcode
barcode_scans.barcode
```

这些字段服务内部生产、工序、测试和追溯。

### 7.2 客户展示优先使用

```text
production_plans.customer_start_barcode
production_plans.customer_end_barcode
work_orders.customer_start_barcode
work_orders.customer_end_barcode
```

这些字段服务客户展示、发货资料和对外查询。

### 7.3 不建议混用

不建议：

- 用客户条码匹配内部工单。
- 用客户条码作为 VFD 测试入口。
- 把客户条码当作 `barcode_relations.sub_barcode`。
- 把内部子条码作为 VFD 测试主入口。

## 8. 推荐校验规则

生产计划：

- 内部开始条码和内部结束条码应成对填写。
- 客户开始条码和客户结束条码应成对填写。
- 客户条码可以为空。

工单：

- 工单内部条码范围应落在生产计划内部条码范围内。
- 同一工厂内，工单内部条码范围不应重叠。
- 同一生产计划下，多个工单内部条码范围不应重叠。
- 工单客户条码范围应落在生产计划客户条码范围内。
- 同一生产计划下，多个工单客户条码范围不应重叠。

VFD 测试：

- 扫描条码必须是内部主条码，格式必须为 `LXVFD11` + 6 位数字。
- 如果扫描到 `LXVFD2` 开头的子条码，应提示“当前扫描的是子条码，请扫描变频器内部主条码”。
- 扫描条码必须能唯一匹配一个工单。
- 如果匹配不到工单，正常生产模式下不允许继续测试。
- 如果匹配到多个工单，必须报错并阻止测试。
- 离线调试模式可以允许不匹配工单，但追溯数据必须标记为调试数据。

## 9. 后续实施建议

一期：

- 保留现有 `barcode_relations` 主子条码设计。
- 内部条码采用 `LXVFD + 角色位 + 部件类型位 + 6 位流水号`。
- 使用内部主条码作为 VFD 测试扫码入口。
- 使用 `start_barcode/end_barcode` 定位工单和生产计划。
- 使用 `customer_start_barcode/customer_end_barcode` 做客户展示范围。
- 新增 `BarcodeRangeMatcher` 服务封装条码解析和范围匹配。

二期：

- 在客户条码生成或导入规则明确后，设计 `barcode_mappings` 单件映射表。
- 支持内部主条码和客户条码互查。
- 支持客户条码查询生产、测试、质量、发货追溯。

## 10. 最终约定

```text
VFD 扫码入口 = 变频器内部主条码
正式内部主条码格式 = LXVFD11 + 6 位数字
正式内部子条码格式 = LXVFD2 + 部件类型位 + 6 位数字
内部主条码范围 = start_barcode / end_barcode
客户条码范围 = customer_start_barcode / customer_end_barcode
内部主子条码关系 = barcode_relations
内部主条码与客户条码映射 = 后续 barcode_mappings
```

这套设计既保留原有主条码/子条码体系，又能支持客户条码一一对应和后续对外查询。
